ТЗ: Backend + Frontend (Django / Vue3)
О задаче
Мини-модуль CRM для работы чатеров с фанами.
Стек
Backend: Django + DRF + Channels (WebSocket)
Frontend: Vue 3 (Composition API) + Pinia
Redis
БД: PostgreSQL (можно SQLite локально)
Контекст
Чатеры ведут переписку с фанами от лица модели. У одного чатера одновременно несколько активных диалогов. Тимлид в реальном времени следит за нагрузкой чатеров и за тем, где фаны ждут ответа слишком долго.
Что нужно реализовать
1. Рабочее место чатера
Список диалогов (фанов) с моделью: последнее сообщение, время, счётчик непрочитанных.
Окно переписки: история сообщений, отправка обычного сообщения и PPV (платного - с ценой).
Сообщения приходят и уходят через WebSocket в реальном времени.
История переписки грузится через REST с пагинацией (подгрузка старых при скролле вверх).
Счётчик непрочитанных обнуляется, когда чатер открыл диалог.
2. Монитор тимлида
Список чатеров онлайн.
По каждому: сколько диалогов в работе и сколько фанов ждут ответа дольше X минут (просрочка).
Обновляется в реальном времени.


3. Метрика времени ответа
Фан написал → запускается таймер ожидания.
Чатер ответил → таймер останавливается.
Диалоги с просрочкой (> X минут без ответа) подсвечиваются у тимлида.
4. Надёжность соединения
При разрыве WebSocket - переподключение и догрузка пропущенных сообщений без дублей и потерь.
Требования к архитектуре
Чёткое разделение: REST - для истории и списков, WebSocket - для live-событий. Не нужно всё гнать через сокеты или всё поллить.
Channels-группы организованы осмысленно (по диалогу / по чатеру / по модели).
Транзакции там, где изменение состояния сопровождается рассылкой события.
Запуск и проверка
Решение должно подниматься одной командой и быть проверяемым.
docker-compose up (или эквивалент) поднимает: бэкенд, Redis, БД, фронт.
В README - раздел «Запуск»: команда, какие порты, как зайти.
Сидер/фикстуры: команда, которая создаёт тестовые данные - 2-3 модели, 3-4 чатера, 1 тимлид, несколько диалогов с историей сообщений. 
Тестовые учётки (логин/пароль) для входа как чатер и как тимлид - прямо в README.
Простой способ сэмулировать входящее сообщение от фана (кнопка/скрипт/эндпоинт) - чтобы realtime можно было проверить в одиночку.
Порог просрочки (X) и grace period для presence сделай короткими и настраиваемыми (env/конфиг).
